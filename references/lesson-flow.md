# 单课节奏与高概念密度处理

## 先选课型，再写课程

不要所有课程都机械套一个模板。先判断本节课属于哪种类型，再从 `templates/lesson-types/` 选择最接近的模板：

| 课型 | 适用场景 | 推荐模板 |
|---|---|---|
| 概念入门课 | 用户第一次接触某个核心概念 | `concept-primer.md.template` |
| 源码调用链课 | 要读懂一条具体代码路径、API、CLI、UI 入口 | `code-trace.md.template` |
| 架构拆解课 | 要理解模块职责、组件关系、设计取舍 | `architecture-walkthrough.md.template` |
| 论文阅读课 | 要读懂论文的问题、方法、实验、局限 | `paper-reading.md.template` |
| 练习/错题复盘课 | 要通过题目/练习暴露误区并迁移 | `exercise-review.md.template` |
| 动手实现课 | 要把理解转成可运行 demo 或功能 | `implementation-lab.md.template` |
| 阶段复盘课 | 阶段收尾、验收、清理问题停车场和复习计划 | `phase-review.md.template` |
| 通用课 | 上述都不完全匹配 | `lesson.md.template` |

## 每节课的推荐模板

1. 本课目标
2. 本课类型
3. 本课在主计划中的位置
4. 总览图
5. 关键概念表
6. 核心流程图 / 架构图 / 调用链图
7. 资料/源码锚点
8. 现实场景类比
9. 常见误区
10. 小练习
11. 理解检查
12. 理解检查评分
13. 反馈纠偏
14. 是否收尾
15. 收尾文档更新

不是每节课都要塞满所有项。概念简单、用户基础好的内容可以压缩成“目标 → 讲解 → 理解检查 → 收尾”四步。判断标准是：这节课的概念密度和用户当前基础是否需要这些辅助结构，而不是机械套模板。

## 理解检查怎么做

不能只问“懂了吗”。让用户做以下任意一种，才算真正检验了理解：

- 用自己的话解释
- 做对照表
- 画流程
- 判断边界（这个方法适用/不适用于哪些情况）
- 设计一个小方案
- 回答场景题
- 指出一个错误做法哪里错了
- 复述源码调用链
- 判断一个改动会影响哪些模块
- 对论文方法做问题-方法-证据-局限复述

## 理解检查 Rubric

评分标准详见 `references/comprehension-rubric.md`。

简化规则：

| 等级 | 名称 | 推进判断 |
|---|---|---|
| 0 | Not Started | 不允许推进 |
| 1 | Exposure | 只算接触过，不允许推进 |
| 2 | Understanding | 默认最低通过线，允许推进 |
| 3 | Skill | 能迁移应用，允许推进 |
| 4 | Capability | 能设计/取舍/独立解决，允许推进 |

只有达到 Understanding 及以上才允许收尾、写 learning-record、写 GLOSSARY、进入下一课。练习复盘、动手实现、架构借鉴等目标，建议最低达到 Skill 或 Capability。

## 高概念密度课程的特殊处理

对于概念多、环节多、抽象度高的内容，不要一口气讲完，默认用：总览图、分层表格、流程图、职责边界图、设计图、小结卡片、企业/生活场景回填，必要时生成图片。

推荐节奏：先讲主干 → 做一次检查 → 再讲下半段 → 再做简化检查 → 最后收尾。把一大块抽象内容拆成两次理解检查，比硬讲完再一次性检查成功率高得多。用户在中途卡住时，继续往下灌新概念只会让断层越滚越大。

## 收尾前检查

收尾前确认：

1. 是否已按 Rubric 记录掌握等级。
2. 是否达到本课最低通过线。
3. 如果只达到 Exposure，是否转为“待理解检查”或“待补课”。
4. 如果达到 Understanding 及以上，是否更新 `learning-records/`、`GLOSSARY.md`、`REVIEW-SCHEDULE.md`。
5. 如果出现旁支问题，是否进入 `QUESTION-PARKING-LOT.md`，而不是直接发散。
