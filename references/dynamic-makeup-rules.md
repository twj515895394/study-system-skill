# 动态补课机制

补课不是临时乱加，是主线推进中发现断层时的正式机制——但必须可控，否则学习会变成无止境的发散。

## 补课必须满足的四个条件

1. 不改变主线方向。
2. 不无限扩展（一次补课通常解决一个具体断层，不要在补课里又发现新的补课需求然后继续往下挖）。
3. 必须说明服务哪个阶段目标——补这个是为了让哪一课能听懂。
4. 补课结束后必须回到主线，明确说"现在回到第 X 课"。

如果发现已经连续插入两次以上补课还没回到主线，停下来跟用户说清楚现状（"我们为了讲清楚 Y，已经插了两节补课，要继续往下补，还是先回到主线，把 Y 当作已知前提往下走"），让用户决定，而不是替用户继续往下挖。

## 补课类型

| 类型 | 触发条件 | 示例 |
|---|---|---|
| 基础补课 | 缺少前置知识 | 先补 TypeScript 类型系统 |
| 概念复习 | 用户说概念太多/跟不上 | 某个抽象层再拆解一次 |
| 源码/资料小课 | 某个点复杂，值得单独细读 | 某段调用链单独细读 |
| 设计练习 | 概念懂了但缺工程感 | 设计一个具体的小功能/方案 |
| 案例课 | 用户要落地、要具体场景 | 结合用户实际场景走一遍 |
| 阶段复盘 | 一个阶段结束 | 阶段验收与复盘 |

记录补课时，在 `PROGRESS.md` / `COURSE-LIST.md` 里标注"是否为补课"，方便事后回看主线被打断的频率和原因——如果某个阶段补课特别多，说明 MASTER-PLAN 里这个阶段的起点判断可能偏高了，值得跟用户一起重新校准。
