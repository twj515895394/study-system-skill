# 理解检查 Rubric

本文件定义理解检查的评分标准。它用于判断用户是否真的达到可以推进下一课的程度，而不是只凭“我懂了”或“讲过了”推进。

## 掌握层级

| 等级 | 名称 | 判定标准 | 是否允许推进 |
|---|---|---|---|
| 0 | Not Started | 用户尚未接触该知识点，或完全说不出它是什么 | 否 |
| 1 | Exposure | 用户听过/看过，能复述名词或表面定义，但不能解释机制、边界或用途 | 否 |
| 2 | Understanding | 用户能用自己的话解释核心机制，并能说出至少一个适用边界或常见误区 | 是 |
| 3 | Skill | 用户能把概念迁移到相似问题，能做对照、改错、流程复述或小范围应用 | 是 |
| 4 | Capability | 用户能基于该知识做设计取舍、方案判断、架构拆解或独立解决实际问题 | 是 |

## 最低通过线

默认最低通过线是 **Understanding（等级 2）**。

只有达到 Understanding 或更高，才允许：

1. 把课程状态更新为 `已完成`。
2. 写入 `learning-records/`。
3. 把术语写入 `GLOSSARY.md`。
4. 把关键知识点加入 `REVIEW-SCHEDULE.md`。
5. 进入下一节主线课。

如果用户只达到 Exposure，不允许收尾，应进入“待理解检查”或“待补课”。

## 理解检查题型选择

| 场景 | 推荐检查方式 | 通过证据 |
|---|---|---|
| 概念入门 | 用自己的话解释 + 举例 | 能解释机制，而不是背定义 |
| 容易混淆的概念 | 对照表 / 边界判断 | 能说出差异、适用条件、反例 |
| 流程/调用链 | 画流程 / 复述链路 | 能讲清输入、处理、输出、关键分支 |
| 源码学习 | 指出关键函数职责 / 改一个小点 | 能定位代码职责和影响范围 |
| 架构学习 | 设计小方案 / 判断取舍 | 能说明为什么这样设计，而不是只说结论 |
| 论文阅读 | 复述问题、方法、实验、局限 | 能区分作者做了什么、没做什么 |
| 练习复盘 | 指出错误原因 + 同类变式 | 能迁移到相似题，而不是只记答案 |

## 评分记录格式

在 lesson、handoff 或 learning-record 中记录理解检查结果时，建议使用：

```md
## 理解检查评分

- 检查方式：场景设计题 / 流程复述 / 代码改错 / 对照表 / 其他
- 用户回答摘要：
- 掌握等级：Exposure / Understanding / Skill / Capability
- 是否通过：是 / 否
- 通过证据：
- 仍存在的问题：
- 后续处理：进入下一课 / 待理解检查 / 待补课 / 需复习
```

## 常见误判

1. **用户说“懂了”不算通过。** 必须有可观察证据。
2. **能背定义不等于 Understanding。** 至少要能解释机制、边界或误区。
3. **做对一道原题不等于 Skill。** 至少要能做一个变式或迁移应用。
4. **讲完一节课不等于可以写 learning-record。** learning-record 只记录已经证明掌握的内容。
5. **回答不完整不一定要补课。** 如果只是表达不清，可以继续追问；如果暴露前置断层，才进入补课。
